  <what-built>
Complete AI Recommendations phase:
- Recommendation SQLAlchemy model and PostgreSQL table (Plan 01)
- Anthropic Claude Sonnet 4.6 primary LLM with forced structured output, Redis 24hr cache, daily call counter, Azure OpenAI fallback (Plan 02)
- FastAPI endpoints: GET /recommendations/, GET /recommendations/summary, POST /recommendations/run (Plan 02)
- APScheduler CronTrigger daily job at 02:00 UTC (Plan 02)
- Frontend RecommendationsPage with summary stats, filter bar, card list, comparison panels, limit banner, empty state (Plan 03)
  </what-built>
  <how-to-verify>
Ensure the application is running (`docker compose up` or local dev server).
Set ANTHROPIC_API_KEY in backend/.env (get from https://console.anthropic.com/settings/keys).

**Step 1 — Page loads**
Navigate to http://localhost:3000/recommendations
Expected: Page loads without error. If no recommendations exist yet, empty state shows with "Generate Recommendations" button (admin users only).

**Step 2 — Manual trigger (AI-01)**
If logged in as admin: click "Generate Recommendations" button (empty state) OR call the API directly:
```bash
curl -X POST http://localhost:8000/api/v1/recommendations/run \
  -H "Authorization: Bearer <your-access-token>"
```
Expected: Returns 202 Accepted. Wait 5-10 seconds, then refresh the page.
Expected: Recommendation cards appear on the page.

**Step 3 — Verify card content (AI-02)**
Inspect one recommendation card. Confirm it shows:
- Resource name (e.g., "vm-prod-eastus")
- Category badge (one of: Right-Sizing, Idle, Reserved, Storage)
- Confidence badge (High/Medium/Low)
- Estimated savings: "Est. savings: $X/mo"
- Confidence percentage: "Confidence: XX%"
- Current → Recommended comparison panel with costs
- Plain-language explanation (2-4 sentences)
- NO "Generated: [date]" label
- NO Apply/Dismiss/Accept buttons

**Step 4 — Verify summary stats**
Check the summary row at the top of the page:
- "Potential Monthly Savings" shows a dollar amount (highlighted green)
- "Total" shows count matching visible cards
- Per-category counts (right-sizing, idle, reserved, storage) sum to total

**Step 5 — Filters work**
Select "Idle" from the Type dropdown.
Expected: Only idle recommendations visible (or empty state if none).
Select "$500+/mo" from Min Savings.
Expected: Only recommendations with savings ≥ $500 visible.
Reset to "All Types" / "Any Savings" — all cards return.

**Step 6 — Redis cache verification (AI-03)**
Trigger recommendations again (POST /run). Check backend logs:
```bash
docker compose logs backend 2>&1 | grep -i "cache hit\|cache"
```
Expected: On the second run for the same resources, log shows "Cache hit" messages (no new LLM calls for resources already processed today).
Alternative verification: Check Redis directly:
```bash
docker compose exec redis redis-cli keys "rec:cache:*"
```
Expected: Keys exist for each resource processed.

**Step 7 — Daily call counter (AI-04)**
Check the /recommendations/summary endpoint:
```bash
curl http://localhost:8000/api/v1/recommendations/summary \
  -H "Authorization: Bearer <your-access-token>"
```
Expected: `calls_used_today` reflects actual calls made, `daily_call_limit` shows 100 (default), `daily_limit_reached` is false (unless limit actually hit).

**Step 8 — Daily limit banner**
To verify the banner without exhausting 100 calls:
Set LLM_DAILY_CALL_LIMIT=1 in backend env, restart backend, trigger generation.
Expected: After 1 LLM call, banner appears: "Daily recommendation limit reached. New recommendations will generate tomorrow."
Reset LLM_DAILY_CALL_LIMIT to 100 after verification.

**Step 9 — Scheduler registered**
Check backend logs on startup:
```bash
docker compose logs backend 2>&1 | grep "recommendation_daily\|apscheduler"
```
Expected: APScheduler logs showing "recommendation_daily" job added with CronTrigger.

**Step 10 — No TypeScript errors**
```bash
cd frontend && npx tsc --noEmit
```
Expected: No output (zero errors).

Create `backend/app/services/anomaly.py` as a new file. Follow the exact same structure and conventions as `cost.py`: async functions, `AsyncSession` parameter, `await session.execute()`, `result.all()` or `.scalars().all()` as appropriate, session committed by the function (or caller — follow the same pattern as `ingestion.py` where `_do_ingestion` commits).

**Function 1: `run_anomaly_detection(session: AsyncSession) -> None`**

This is the main entry point called from `_do_ingestion`. It:
1. Computes a 30-day rolling baseline average daily cost per `(service_name, resource_group)` pair using a two-step subquery (see Research Pattern 1).
2. Guards against empty baseline: if `baseline_rows` is empty (no 30-day history yet), log a warning and return early — do NOT write anomalies.
3. Finds the most recent completed billing day: query `MAX(usage_date)` from `billing_records` rather than hardcoding `today - 1` (handles Azure data latency — Research Open Question 2).
4. Fetches current day spend per `(service_name, resource_group)` for that `check_date`.
5. For each pair: compute `pct_deviation = (current - baseline_avg) / baseline_avg * 100`. Skip if `pct_deviation < 20.0` or `baseline_avg <= 0`.
6. Compute `estimated_monthly_impact = (current - baseline_avg) * 30`. Skip if `< 100` (noise floor).
7. Classify severity: `critical` if `>= 1000`, `high` if `>= 500`, else `medium`.
8. Generate description: `f"Spend increased {pct_deviation:.0f}% in {row.resource_group}"`.
9. Call `await upsert_anomaly(session, ...)` for each flagged pair. Collect the `(service_name, resource_group)` set of still-active anomalies.
10. Call `await auto_resolve_anomalies(session, still_active, check_date)`.
11. Commit once at the end: `await session.commit()`.

**Function 2: `async def upsert_anomaly(session, *, service_name, resource_group, detected_date, baseline_daily_avg, current_daily_cost, pct_deviation, estimated_monthly_impact, severity, description) -> None`**

Use `from sqlalchemy.dialects.postgresql import insert as pg_insert`. Insert into `Anomaly` table. On conflict on `(service_name, resource_group, detected_date)`, update: `baseline_daily_avg`, `current_daily_cost`, `pct_deviation`, `estimated_monthly_impact`, `severity`, `description`, `updated_at`. Do NOT update `status` or `expected` (preserve user actions). Do NOT commit here — caller commits.

**Function 3: `async def auto_resolve_anomalies(session, still_active: set[tuple[str, str]], check_date: date) -> None`**

Query `Anomaly` rows where `status IN ('new', 'investigating')`, `detected_date == check_date`, `expected == False`. For each where `(service_name, resource_group)` is NOT in `still_active`, set `status = 'resolved'` and `updated_at = datetime.now(timezone.utc)`. Do NOT commit here — caller commits.

**Function 4: `async def get_anomalies(session, *, status=None, severity=None, service_name=None, resource_group=None) -> list`**

Query all `Anomaly` rows (no default status filter — full history). Apply optional filters from params. Order by `detected_date DESC`, then `estimated_monthly_impact DESC`. Return `result.scalars().all()`.

**Function 5: `async def get_anomaly_summary(session) -> dict`**

Return a dict with:
- `active_count`: count where `status IN ('new', 'investigating')` AND `expected == False`
- `critical_count`: count where active AND `severity == 'critical'`
- `high_count`: count where active AND `severity == 'high'`
- `medium_count`: count where active AND `severity == 'medium'`
- `total_potential_impact`: sum of `estimated_monthly_impact` where active
- `resolved_this_month`: count where `status == 'resolved'` AND `updated_at` in current calendar month
- `detection_accuracy`: `(total_detected - expected_count) / total_detected * 100` if `total_detected > 0`, else `None`. Here `total_detected` = count of all non-dismissed anomalies ever; `expected_count` = count where `expected == True`.

Use individual `select(func.count(...))` or `select(func.sum(...))` queries, or a single query with `case()` expressions — match the style used in `cost.py` for MTD/MoM aggregation.

**Function 6: `async def update_anomaly_status(session, anomaly_id: uuid.UUID, new_status: str) -> Anomaly | None`**

Load `Anomaly` by id, update `status` and `updated_at`, commit, return the updated object. Return `None` if not found.

**Function 7: `async def mark_anomaly_expected(session, anomaly_id: uuid.UUID) -> Anomaly | None`**

Load `Anomaly` by id, set `expected = True`, `status = 'dismissed'`, `updated_at = now()`, commit, return updated object. Return `None` if not found.

**Function 8: `async def get_anomalies_for_export(session, *, severity=None, service_name=None) -> list`**

Same as `get_anomalies` but returns all columns needed for CSV. Reuse `get_anomalies` or duplicate with explicit column selection. The export endpoint will format this into CSV.

from datetime import date, datetime
from typing import Optional
from uuid import UUID

from pydantic import BaseModel


class AnomalyResponse(BaseModel):

    detected_date: date
    service_name: str
    resource_group: str
    description: str
    severity: str
    status: str
    expected: bool
    pct_deviation: float
    estimated_monthly_impact: float
    baseline_daily_avg: float
    current_daily_cost: float
    created_at: datetime

    critical_count: int
    high_count: int
    medium_count: int

**Step C — Create FastAPI router (`backend/app/api/v1/anomaly.py`):**

Follow the exact same pattern as `backend/app/api/v1/cost.py`. Router prefix `/anomalies`, `get_current_user` dependency on all endpoints. Six endpoints:

1. `GET /` → `list_anomalies` — query params: `status: str | None`, `severity: str | None`, `service_name: str | None`, `resource_group: str | None`. Returns `list[AnomalyResponse]`. Calls `get_anomalies(db, status=status, severity=severity, service_name=service_name, resource_group=resource_group)`. Maps ORM objects to `AnomalyResponse` using `AnomalyResponse.model_validate(row)`.

2. `GET /summary` → `anomaly_summary` — Returns `AnomalySummaryResponse`. Calls `get_anomaly_summary(db)`. Maps dict to `AnomalySummaryResponse(**summary_dict)`.

3. `PATCH /{anomaly_id}/status` → `update_status` — Body: `AnomalyStatusUpdate`. Validates `body.status` is one of `['investigating', 'resolved', 'dismissed']` — raise `HTTPException(400)` if invalid. Calls `update_anomaly_status(db, anomaly_id, body.status)`. Returns `AnomalyResponse` of updated record. Raises `HTTPException(404)` if not found.

4. `PATCH /{anomaly_id}/expected` → `mark_expected` — No body needed. Calls `mark_anomaly_expected(db, anomaly_id)`. Returns `AnomalyResponse`. Raises `HTTPException(404)` if not found.

5. `GET /export` → `export_anomalies` — Query params: `severity: str | None`, `service_name: str | None`. Returns `StreamingResponse` with CSV. Calls `get_anomalies_for_export(db, severity=severity, service_name=service_name)`. Uses `io.StringIO` + `csv.writer`. CSV columns: `detected_date, service_name, resource_group, severity, status, pct_deviation, estimated_monthly_impact, description`. Header: `Content-Disposition: attachment; filename="anomaly-report.csv"`, `Content-Type: text/csv`. Use `StreamingResponse(iter([output.getvalue()]), media_type='text/csv', headers={...})` — same pattern as cost.py export.

6. `GET /services` → `list_services` — Returns `list[str]` of distinct `service_name` values from the `anomalies` table. Used to populate the Service filter dropdown in the frontend. Query: `SELECT DISTINCT service_name FROM anomalies ORDER BY service_name`.

7. `GET /resource-groups` → `list_resource_groups` — Returns `list[str]` of distinct `resource_group` values. Same pattern as `/services`.

Note: endpoints 6 and 7 are bonus convenience endpoints for the frontend filter dropdowns. If adding them would complicate the router, use a single `GET /filter-options` endpoint returning `{ services: [], resource_groups: [] }` instead — Claude's discretion.
